# Session 2026-06-14 — Bedroom blind BLE reliability fix

## Problem

The bedroom blind (TS5001, MAC `FD:74:46:D2:50:0F`) was almost entirely unresponsive to commands — neither from HA automations nor from the HA UI. The device was repeatedly connecting and disconnecting every ~5 seconds in the logs.

The living room blind (TS2900, MAC `EC:A5:21:CA:9A:F8`) was working fine throughout.

## What we tried (in order)

### 1. Connect-on-demand (`auto_connect: false`)
Added a `ble_client.connect` / `ble_client.disconnect` around each cover action instead of maintaining a persistent connection. Hypothesis: maybe the blind doesn't like being held open permanently.

**Result: worse.** Commands still failed and reliability dropped further. Abandoned quickly.

### 2. Persistent connection + CONNECTION_MESSAGE only (no timestamp)
Switched back to persistent (`auto_connect: true`). Sent the connection message (`ff 03 03 03 03 78 78 78 78 78 78`) on `on_connect` — same as the living room config.

**Result: still disconnecting every ~5 seconds.** Logs showed repeated `on_connect` / `on_disconnect` cycles. No commands got through. Disconnect reason was `0x13` = `HCI_ERR_REMOTE_USER_TERM_CONN` — the **blind itself** was terminating the connection.

### 3. Persistent connection + CONNECTION_MESSAGE + timestamp (fix)

Found the root cause in [pink88/Tuiss2HA](https://github.com/pink88/Tuiss2HA) (`const.py`): the TS5001 requires **both** a connection message AND a timestamp write on every fresh connection. Without the timestamp the blind times out and severs the connection after ~5 seconds.

The TS2900 (living room) is more lenient — it only needs the connection message once at boot.

Sent both on every `on_connect`:
1. CONNECTION_MESSAGE: `ff 03 03 03 03 78 78 78 78 78 78`
2. Timestamp: `ff 78 ea 41 02 00 YY MM DD HH mm SS` (using `sntp_time.now()`)

Set `br_ready = true` only **after both writes complete**.

**Result: working.** Blind connects, stays connected, responds to all commands.

## Key bugs encountered

### `wait_until` silently hangs in cover template actions (ESPHome 2026.3)

Commands from HA arrived (entity showed available, web server buttons present) but cover `open_action` / `close_action` / `position_action` never executed — zero log output from inside the actions.

Root cause: `wait_until` used to wait for `br_ready` before issuing BLE writes. It silently hangs inside cover template action sequences in ESPHome 2026.3.

Fix: replaced every `wait_until` with a direct `if(br_ready)` check. Commands that arrive during the brief reconnect window are dropped and logged at WARN level.

### PlatformIO build cache collision

After making firmware changes, flashing would silently succeed but the old firmware stayed. Fixed by clearing the ESPHome build cache before reflashing:
```bash
docker exec esphome rm -rf /config/.esphome/build/bedroom-esp32
docker exec esphome esphome run /config/bedroom-esp32.yaml --device 192.168.0.142
```

### `br_initialized` guard (removed for bedroom)

The living room config has a `lr_initialized` global that prevents re-sending the connection message after the first boot (since TS2900 only needs it once). The bedroom originally copied this.

Removed it for bedroom: the TS5001 needs the full handshake on **every** connection — the guard was preventing the timestamp from being sent after reconnects, causing the 5-second disconnect loop to repeat.

## Current firmware state

### bedroom-esp32 (`bedroom-esp32.yaml`)

- Persistent BLE connection (`auto_connect: true` by default in ESPHome)
- `on_connect`: 500ms delay → CONNECTION_MESSAGE → 200ms delay → timestamp → 200ms delay → `br_ready = true`
- `on_disconnect`: `br_ready = false`
- All cover actions guard on `br_ready`; dropped commands log at WARN: `"Bedroom blind: open dropped (not connected)"`
- `binary_sensor.all_doors_and_windows_secure` — connectivity sensor for HA automations
- SNTP time component required for the timestamp lambda

### living-room-esp32 (`living-room-esp32.yaml`)

- Persistent BLE connection
- `lr_initialized` global: CONNECTION_MESSAGE sent only on first `on_connect`
- No timestamp required (TS2900 does not need it)
- Cover actions write directly without `br_ready` guard (TS2900 tolerates this)
- `lr_connected` binary sensor

## Open questions / next steps

- The bedroom blind does still briefly disconnect overnight occasionally — unclear if this is BLE range, blind firmware, or power saving. Worth monitoring.
- Living room and bedroom configs have diverged significantly (different models, different handshake requirements). Keep them as separate files — do not try to merge into a shared package.
- If the 5-second disconnect loop ever comes back: first check Seq for whether the timestamp write is happening. If it is, check if SNTP has synced (`sntp_time.now().year` should be > 2000 — if it's 1970 the NTP hasn't synced yet and the timestamp will be wrong).
